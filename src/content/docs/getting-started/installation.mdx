## Prerequisites

Before installing SCC, ensure you have:

| Requirement | Minimum Version | Check Command |
|-------------|-----------------|---------------|
| Python | 3.10+ | `python --version` |
| uv (recommended) | Latest | `uv --version` |
| Container runtime | Docker Engine 20.10+ | `docker --version` |
| Git | 2.30+ | `git --version` |

### Container Runtime

SCC requires a Docker-compatible container runtime. Any of these work:

| Runtime | Platform | Notes |
|---------|----------|-------|
| [Docker Engine](https://docs.docker.com/engine/) | Linux, macOS, Windows | Standard Docker daemon |
| [OrbStack](https://orbstack.dev/) | macOS | Fast, lightweight Docker alternative |
| [Colima](https://github.com/abiosoft/colima) | macOS, Linux | Open-source Docker runtime |
| [Docker Desktop](https://www.docker.com/products/docker-desktop/) | macOS, Windows, Linux | Full GUI experience |
| [Podman](https://podman.io/) | Linux, macOS | 🔄 Planned — not yet fully validated |

SCC auto-detects the available runtime. **Docker Desktop is not required.**

<Aside type="caution">
  **A container runtime must be installed and running** before using SCC. Ensure the daemon is active (`docker info` should succeed).
</Aside>

## Verify Installation

<Steps>
1. **Check SCC is installed**

   ```bash
   scc --version
   ```

2. **Run diagnostics**

   ```bash
   scc doctor
   ```

   This checks:
   - Container runtime is running and accessible
   - Runtime version meets requirements
   - Git is configured
   - Config directories exist
   - Provider image availability and auth readiness

3. **Check a specific provider**

   ```bash
   scc doctor --provider codex
   ```

   If a provider image is missing, doctor shows the exact build command.

4. **Review any warnings**

   If `scc doctor` shows issues, address them before proceeding.
</Steps>

## Platform-Specific Notes

<Tabs syncKey="os">
  <TabItem label="macOS">
  - Install Docker Engine (via OrbStack or Colima) or Docker Desktop
  - OrbStack is recommended for speed and low resource usage
  - Ensure the daemon is running (`docker info`)
  - No additional configuration needed
  </TabItem>
  <TabItem label="Linux">
  - Docker Engine is the standard choice
  - Ensure your user is in the `docker` group:
    ```bash
    sudo usermod -aG docker $USER
    # Log out and back in for changes to take effect
    ```
  </TabItem>
  <TabItem label="Windows (native)">
  - Windows Terminal, PowerShell, Git Bash, and VS Code terminal are supported
  - Install Docker Desktop (WSL2 backend recommended) and keep it running
  - Use `uv` for the most reliable install and upgrades
  </TabItem>
  <TabItem label="Windows (WSL2)">
  - Run SCC inside WSL2 for a Linux-like environment
  - Install Docker Desktop with WSL2 backend and enable WSL integration
  - File operations are fastest inside the WSL filesystem (`~/projects`)

  <Aside type="tip">
    For best performance, keep your projects inside the WSL filesystem instead of `/mnt/c/`.
  </Aside>
  </TabItem>
</Tabs>

## Container Images

SCC builds provider-specific container images on first use:

| Image | Purpose |
|-------|---------|
| `scc-agent-claude` | Claude Code agent with safety wrappers |
| `scc-agent-codex` | Codex agent with safety wrappers |
| `scc-base` | Shared base image with safety evaluator |
| `scc-egress-proxy` | Squid proxy for `web-egress-enforced` policy |

The first `scc start` with a provider triggers an automatic image build (takes a few minutes). Subsequent launches use the cached image.

**Manual build** (if needed):

```bash
docker build -t scc-agent-claude images/scc-agent-claude/
docker build -t scc-agent-codex images/scc-agent-codex/
```
