Complete command reference for SCC (Sandboxed Coding CLI).

<Aside type="note">
  The daily workflow is usually simple: run `scc setup` once to connect Claude, Codex, or both, then use `scc start` or just `scc` to launch sessions. Use `scc provider show` and `scc provider set` when you want to inspect or change how SCC chooses between providers.
</Aside>

## Global Options

These options are available on all commands:

| Option | Description |
|--------|-------------|
| `--debug` | Show detailed error information |
| `--version`, `-v` | Show version and exit |
| `--help`, `-h` | Show help for any command |

## Exit Codes

All SCC commands use consistent exit codes.
Human output and `--json` output use the same table.

| Code | Constant | Meaning |
|------|----------|---------|
| 0 | `EXIT_SUCCESS` | Operation completed successfully |
| 1 | `EXIT_NOT_FOUND` | Target not found |
| 2 | `EXIT_USAGE` | Invalid usage/arguments |
| 3 | `EXIT_CONFIG` | Configuration or network error |
| 4 | `EXIT_TOOL` | External tool failed (git, docker) |
| 5 | `EXIT_PREREQ` | Prerequisites not met |
| 6 | `EXIT_GOVERNANCE` | Blocked by governance policy |
| 130 | `EXIT_CANCELLED` | User cancelled (Esc/Ctrl+C) |

### Using Exit Codes in Scripts

```bash
scc start --dry-run --non-interactive --provider claude --team ci .
exit_code=$?

case $exit_code in
  0) echo "Config valid" ;;
  3) echo "Config error" ;;
  5) echo "Missing prerequisites" ;;
  6) echo "Blocked by governance" ;;
  *) echo "Other error: $exit_code" ;;
esac
```

## JSON Output

Commands with `--json` support output a standardized envelope:

```json
{
  "apiVersion": "scc.cli/v1",
  "kind": "CommandName",
  "metadata": {
    "generatedAt": "2025-12-23T10:00:00Z",
    "cliVersion": "1.7.3"
  },
  "status": {
    "ok": true,
    "errors": [],
    "warnings": []
  },
  "data": { }
}
```

### Using JSON Output

```bash
# Pretty print
scc status --json --pretty

# Parse with jq
scc start --dry-run --json | jq '.data.plugins'

# Check status in scripts
if scc org validate config.json --json | jq -e '.status.ok'; then
  echo "Config is valid"
fi
```

<Aside type="tip">
  Use `--pretty` for human-readable formatting during development, omit it for machine parsing.
</Aside>

## Command Categories

| Category | Commands | Purpose |
|----------|----------|---------|
| [Essential](/reference/cli/essential/) | `setup`, `doctor`, `start`, `stop`, `status` | Core workflow |
| [Provider](/reference/cli/provider/) | `provider show`, `provider set` | Provider management |
| [Sessions](/reference/cli/sessions/) | `sessions`, `sessions prune`, `context` | Session management |
| [Worktree](/reference/cli/worktree/) | `worktree *` | Parallel development |
| [Team](/reference/cli/team/) | `team *` | Team profiles |
| [Profile](/reference/cli/profile/) | `profile *` | Personal project profiles |
| [Config](/reference/cli/config/) | `config`, `config paths`, `statusline`, `update` | Configuration |
| [Dev Environment](/reference/cli/dev/) | `dev run` | Approved host-owned project support actions |
| [Governance](/reference/cli/governance/) | `exceptions`, `unblock`, `audit` | Security & policies |
| [Organization](/reference/cli/organization/) | `org *` | Org admin commands |
| [Admin](/reference/cli/admin/) | `reset`, `prune`, `stats`, `support`, `completion` | Maintenance & diagnostics |

## Getting Help

```bash
# General help
scc --help

# Command-specific help
scc start --help
scc provider set --help
scc worktree create --help

# Debug mode for troubleshooting
scc --debug start ~/project
```
