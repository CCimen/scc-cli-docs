SCC v1 focuses on a safe-by-default provider-neutral runtime, governed configuration, and clear explainability.
This page defines what SCC enforces at runtime, what is advisory, and what is out of scope in v1.

## Core identity

- Runs AI coding agents (Claude Code, Codex) inside OCI container sandboxes with controlled mounts.
- Distributes org and team configuration consistently across users, projects, and providers.
- Enforces governance where SCC controls the surface.
- Explains what is active, blocked, or denied and why.
- Provider-neutral: same governance, safety, and config pipeline regardless of which agent you use.

## Supported providers

- **Claude Code** — first-class provider with full adapter support
- **Codex** — first-class provider with full adapter support
- Additional providers (e.g., OpenCode) are architecturally supported but not yet shipped

## Non-goals for v1

- No data loss prevention (DLP) guarantees.
- No inspection of plugin internals (plugins are the trust unit).
- No raw TCP/UDP filtering beyond HTTP(S) in web-egress-enforced mode.
- No validated Podman support.

## Enforcement states

- **Enforced:** SCC changes runtime behavior deterministically.
- **Partially enforced:** SCC enforces part of the intent and warns about the rest.
- **Advisory:** SCC validates and reports but does not enforce at runtime.
- **Out of scope:** SCC does not attempt to govern this surface.

## Scope by surface

| Surface | Status | Notes |
| --- | --- | --- |
| Plugins | Enforced | Managed via marketplaces and injected into provider-native runtime settings. |
| Marketplaces | Enforced | Materialized and injected into runtime settings. |
| MCP servers (org/team/project) | Enforced | Injected into runtime settings after policy gates. |
| MCP servers (.mcp.json) | Advisory | SCC does not modify repo files in v1. |
| MCP servers (plugin-bundled) | Out of scope | Block the plugin to block bundled MCPs. |
| network_policy: `open` | Enforced | No restriction — unrestricted egress. |
| network_policy: `web-egress-enforced` | Enforced | Topology-based isolation: internal Docker network + Squid proxy sidecar. HTTP/HTTPS over IPv4 only. Raw TCP/UDP not filtered. |
| network_policy: `locked-down-web` | Enforced | `--network=none` — no external access. |
| Built-in safety engine | Enforced | Fail-closed shell wrappers for git + network tools. Provider-neutral. |
| safety_net plugin policy | Enforced when enabled | Enforced by the scc-safety-net plugin via agent-native hooks. |
| session.auto_resume | Advisory | Accepted in config but not enforced yet. |
| Provider selection | Enforced | Unknown or forbidden providers fail closed — never silently fall back. |

<Aside type="note">
  `web-egress-enforced` is real topology enforcement — the agent container physically cannot bypass the proxy because it's on an internal-only Docker network. This is not just proxy env var injection.
</Aside>

## Plugin model truthfulness

Claude Code and Codex have different native plugin, rules, and hook surfaces. SCC does **not** pretend they share an identical plugin format. Instead:

- SCC unifies governance at the **approval/bundle level** — what's allowed, blocked, or required
- Provider-specific **renderers** project the approved bundle into each provider's native configuration
- The same org security blocks apply to both providers, but the enforcement mechanism differs per provider
